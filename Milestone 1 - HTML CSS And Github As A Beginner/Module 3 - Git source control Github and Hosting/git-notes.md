### গল্পের মতো করে গিট এবং গিটহাব শিখুন নিজের মাতৃভাষা বাংলায়

আজকে আমরা আলোচনা করবো গিট এবং গিটহাব নিয়ে। তবে প্রথমেই মনে রাখতে হবে যে গিট এবং গিটহাব দুটি ভিন্ন জিনিস কিন্তু রিলেটিভ। আপাতত বোঝার জন্য ধরে নিতে পারেন গিট এবং গিটহাব হলো কফি এবং কফিশপ এর মতো। একটু পরেই বিস্তারিত বলছি, তখন সব কনফিউশন দূর হয়ে যাবে। আপাতত গিট কি? সেটা বুঝে আসি চলুন।

গিট কি?
গিট সব সময় আপনার কোডের দিকে তাকিয়ে থাকে, that’s it। ২৪ ঘণ্টা, ৩৬৫ দিন আপনার কোডের দিকে নজর রাখে। আপনার কোডে কী পরিবর্তন আসছে, সেটার দিকে নজর রাখে এবং প্রতিটা পরিবর্তনের একটি করে রেকর্ড রাখে। এটা এমন না যে শুধু আপনার কোডের কথা বলছি এখানে বুঝানো হয়েছে যে ফাইলের মধ্যে যা কিছু আছে সব ইমেজ, টেক্সট, এভরিথিং। এমন কি কে চেঞ্জ করছে সেটার ও রেকর্ড রেখে দেয়। which is really something cool about git right?

কিন্তু আমার কাছে এটা মনে হয় না, আমার কাছে যেটা বেস্ট ফিচার মনে হয় সেটা হলো এটা ফাইলের একেকটা চেঞ্জের ভার্শন রেখে দেয়। যেমন ধরেন আপনার ক্লাইন্টকে প্রোজেক্ট ডেলিভারি দিয়ে দিছেন। ১ মাস পর ক্লাইন্ট এসে আপনাকে একটা চেঞ্জ দিলো, তো আপনি তার কথা অনুযায়ী চেঞ্জ করে দিলেন। কিন্তু তার কিছু দিন পর এসে বললো যে ভাই আগের ভার্শনটাই ভালো ছিল। এখন তো আপনি বিপদে পড়ে গেলেন। আপনাকে আবার কোড রিমুভ করে আগের জাইগায় নিয়ে যেতে হবে, যেটা পেইনফুল। এই সমস্যা সমাধানের জন্নই গিট এসেছে। এজন্যই গিটকে ভার্শন কন্ট্রোল সিস্টেম বলা হয়।

গিটহাব কি ?
ছোট্ট একটা গল্প বলি চলেন। এনামুল আর আহাদুল ২জন বন্ধু। তারা সিদ্ধান্ত নিয়েছে যে আজকে একটা ওয়েবসাইট তৈরি করবে। একজন হেডার আর নাভবারের পার্ট করবে, আরেকজন ফুটারের পার্ট করবে। যেই ভাবা সেই কাজ—তারা কোড করা শুরু করলো। একটু পর এনামুলের কোড করা শেষ, সে বসে আছে আহাদুলের কখন শেষ হবে।

একটা পর্যায়ে আহাদুলেরও শেষ হলো। এখন এনামুল আহাদুলকে বাসায় ডাক দিলো আহাদুলের কোড নেওয়ার জন্য, কারণ তার কাছে তো আহাদুলের কোড নেই। তারপর আহাদুল তার পেনড্রাইভে করে কোড নিয়ে যায়। পরে সেই কোড পেনড্রাইভ থেকে এনামুলের পিসিতে নিয়ে হেডার আর নাভবারের নিচে বসিয়ে দিলো। এভাবেই তাদের প্রজেক্ট সম্পূর্ণ হলো।

এখন আমাকে বলেন, এখানে আসলে ঘটনা কী ঘটলো? এখানে মূলত ২টা জিনিস মনে রাখতে হবে:
১। এনামুল যখন কোড করছিল, তখন তার কোড শেষ হওয়ার পর সে আহাদুলের জন্য বসে ছিল, কখন তার কাজ শেষ হবে।
২। আহাদুল তার কোড এনামুলকে দেওয়ার জন্য পেনড্রাইভে করে নিয়ে এসেছিল।

আপনি ভাবতে পারছেন, এখানে কী সমস্যা হচ্ছে? বিশাল সমস্যা। এই যে একজন ডেভেলপারকে অন্যজনের কোডের জন্য বসে থাকতে হয় বা একজনের কোড আরেকজনের কোডের সঙ্গে এড করার জন্য যেই পেইনফুল কাজ করতে হয়—এটা যেন না করা লাগে, সেটার জন্যই এমন একটা সেন্ট্রাল জায়গা দরকার, যেখানে একজন ডেভেলপারের কাজ শেষ হয়ে গেলে তার কোড পাঠিয়ে দেবে, আর বাকি জন তার কাজ শেষ হলে সেও পাঠিয়ে দেবে। পরে ২ জনের কোড মিলিয়ে প্রজেক্ট সম্পূর্ণ করা হবে।

এই যে একটা সেন্ট্রাল জায়গার কথা বললাম—এই কাজটাই করে গিটহাব।
গিটহাব মূলত এমন একটা সেন্ট্রাল জায়গা যেখানে ডেভেলপাররা তাদের কোড ডেপলয় বা রেখে দিতে পারে অনলাইনে কাজের সুবিধার জন্য।
আশা করছি গিট এবং গিটহাব নিয়ে একদম রুট লেভেলের ধারনা পেয়ে গেছেন। কেও আপনাকে এখন প্রশ্ন করলে তাকে সুন্দর করে বুঝিয়ে দিতে পারবেন।

এবার আসি আমরা এই কাজ গুলো করবো কিভাবে

Git Architecture:

Git নিয়ে কাজ করতে হলে আগে আমাদের Git এর concept বুঝতে হবে। উপরে তো শুধু বললাম যে Git কী? এখন একটু deep dive দেখি। চলুন!

Git এর ক্ষেত্রে আমরা দুটি বিষয় নিয়ে চিন্তা করব—একটি হলো local, আরেকটি হলো remote। Local হলো সেই জায়গা যেখানে আমরা কাজ করব, মানে আমাদের লোকাল মেশিনে। আমাদের সমস্ত কাজকর্ম এই লোকাল জায়গায় থাকবে।

আরেকটি হলো remote, যেটি ক্লাউডে থাকে। এক্ষেত্রে আমরা GitHub-কে বেছে নিতে পারি।

Local-এ আমরা যে ফোল্ডারে কাজ করি, সেটাকে বলা হয় working directory। যখন লোকাল অবস্থায় আমাদের মনের মাধুরী মিশিয়ে কাজ টাজ শেষ করব, তখন সেটাকে stage-এ তুলে দেব। মানে, আমার কাজ শেষ, এখন তুমি stage-এ কোড নিয়ে যেতে পার, এটি এখন ready।

এরপর stage থেকে আমরা local repository-তে তুলব। এখন আমরা যদি চাই, এই কোড অন্য কোনো মেশিনে পাঠাতে বা ধরুন অন্য কাউকে দিতে, তখন এই কোডকে remote-এ নিয়ে যাব, মানে GitHub-এ আপলোড করব। এইটুকুই হলো Git-এর concept।

এখন আমরা একটু প্র্যাক্টিক্যাল ভাবে কোড করে দেখি উদাহারণ

পথমে আমরা একটি ফোল্ডার তৈরি করব, যার নাম হবে git-example। এর মধ্যে ২টি ফাইল এবং ১টি ফোল্ডার থাকবে। ফাইল ২টির নাম হবে example1.txt এবং example2.txt। ফোল্ডারটির নাম হবে text, যার মধ্যে একটি ফাইল থাকবে, যার নাম হবে example3.txt।

ফোল্ডারের স্ট্রাকচার হবে নিচের মতো:
git-example/
├── example1.txt
├── example2.txt
└── text/
└── example3.txt

আমাদের git-example ফোল্ডারটি এখন আমাদের working directory। এই ফোল্ডারের মধ্যে আমরা যে ফাইলগুলো নিয়েছি, সেগুলোর প্রত্যেকটিতে গিয়ে আমরা কিছু পরিবর্তন করলাম। ধরেন, example1.txt ফাইলটিতে গিয়ে লিখলাম "this is example1 file", এবং এভাবেই বাকিগুলোতেও পরিবর্তন করে এলাম "this is example2 file", "this is example3 file"।

এখন যদি আমরা চাই যে Git আমাদের এই git-example ফোল্ডারটিকে নজর রাখুক বা track করুক, তাহলে খুব সহজে আমরা এই ফোল্ডারের ভিতরে গিয়ে terminal খুলে নিচের কমান্ডটি দিতে পারি:

git init

এই git init কমান্ডটি দেয়ার মানে হলো, আমাদের এই প্রজেক্টে Locally Git ইতিমধ্যেই ইনিশিয়ালাইজ (initialize) হয়ে গেছে। এর ফলে, এই ফোল্ডারটি এখন থেকে একটি Git repository হয়ে যাবে এবং আমরা এই প্রজেক্টের সব পরিবর্তন Git দিয়ে ট্র্যাক করতে পারবো।

এখন যদি আমরা `example2.txt` ফাইলটিতে গিয়ে কিছু পরিবর্তন করি, যেমন ফাইলের শেষে **"update1"** যোগ করি, তাহলে ফাইলের কনটেন্ট হয়ে যাবে:

```
this is example2 file update1
```

এখন আমরা **terminal**-এ গিয়ে নিচের কমান্ডটি চালাব:

```bash
git status
```

এই কমান্ডটি চালানোর পর আমরা এমন একটি আউটপুট দেখতে পাব:

```
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)

        modified:   example2.txt
```

এই **`modified:   example2.txt`** বলছে যে `example2.txt` ফাইলটি পরিবর্তিত (modified) হয়েছে। অর্থাৎ, **Git** আমাদের ফাইলের পরিবর্তনগুলো নজর রাখছে এবং জানিয়ে দিচ্ছে।

আমরা এতক্ষণ working directory কাজ করলাম এখন এটাকে stage এ তুলবো। আপনি এটা করার জন্য কয়েকটা কমান্ড আছে ওদিকে যাচ্ছি না। একটা জিনিস অনেক ভাবে করা যায় আমি সহজ টা বলে দিচ্ছি

```
git add .
```

এখানে . এর মানে হলো আপনার current folder-এ যা যা আছে, সবকিছুকে stage-এ তুলবে। এখন যদি আপনি git status চেক করেন, তাহলে দেখতে পাবেন যে সবকিছু stage-এ উঠে গেছে।
apni arokom akta msg paben .
============================= image asbe

যদি আপনি সবকিছু আগের অবস্থায় ফিরে যেতে চান, তাহলে git reset কমান্ড ব্যবহার করতে পারেন। এই কমান্ডটি stage থেকে সবকিছু আবার ফিরে নিয়ে আসবে এবং টোকেন থাকবে না।

এভাবে করতে হবে:

```bash
git reset
```

এটি ব্যবহার করার পর stage থেকে সবকিছু সরিয়ে নেওয়া হবে, কিন্তু আপনার ফাইলগুলোতে কোনো পরিবর্তন আসবে না, অর্থাৎ ফাইলগুলো working directory-এ আগের মতোই থাকবে।

এখন আমাদের কোড লোকাল থেকে stage আছে, এবার দেখবো যে কিভাবে আমরা stage থেকে Local Repository তে নিতে পারি। এটাকে গিটের ভাষায় বলা হয় কমিট করা মানে সেভ করা।

আপনি এখন টার্মিনালে গিয়ে এই কমান্ডটি দিতে পারেন:

```
git commit -m "i have made some changes to files"
```

এটি দিয়ে আপনি ফাইলগুলোকে commit করে ফেলেছেন। এখন যদি আপনি git status চেক করেন, তাহলে আপনি দেখতে পাবেন:

```
nothing to commit, working tree clean
```

এর মানে হলো, আপনার সবকিছু commit হয়ে গেছে এবং এখন আপনার Local Repository-এ কোড সেভ হয়ে গেছে।

NOTE: যদি কখনো এমন পরিস্থিতি আসে, যেখানে আপনি commit করার পরও চাইছেন যে আপনার কোনো কিছু প্রয়োজন নেই বা আপনি আগের অবস্থায় ফিরে যেতে চান, তখন আপনি git reset --hard কমান্ড ব্যবহার করতে পারেন:

```
git reset --hard
```

এই কমান্ডটি ব্যবহার করলে আপনার commit হয়ে যাওয়া পরিবর্তনগুলো বাতিল হয়ে যাবে, এবং working directory এবং staging area উভয়ই আগের অবস্থায় ফিরে যাবে।

### Branching, Checkout, & Merging

branch শব্দের অর্থ হচ্ছে শাখা প্রশাখা, এটা আপনারা সবাই জানেন গিটে এমনটাই বুঝায়। ডিফল্ট অবস্থায় শুধু main নামের একটি branch থাকে। আমরা branch কেন তৈরি করি, ধরুন ওই যে উপরে এনামুল আর Ahadul এর কথা বললাম, তাদের মধ্যে এনামুল বললো যে, "দোস্ত, চল আমাদের ওয়েবসাইটে নতুন একটা ফিচার অ্যাড করি", আর Ahadul বললো, "ওয়েবসাইটে যদি পরে কোনো সমস্যা হয়?"

তখন এনামুল বললো, "তাহলে আমরা একটা কাজ করি, আমরা এই ফিচারের জন্য নতুন করে একটা branch তৈরি করি, কাজ শেষ হলে পরে সেটাকে main branch এ নিয়ে আসবো, তখন আর main ওয়েবসাইটে সমস্যা হবে না।"

এখন এই অবস্থায় নতুন ফিচার এড করতে এনামুল কে যেই যেই স্টেপ এর মধ্যে দিয়ে যেতে হবে টা হলো:

1. New Branch Create:
   নতুন ব্রাঞ্চ তৈরি করার জন্য এই কমান্ড ব্যবহার করবে:

git branch new-feature

এখানে new-feature হল নতুন ফিচারের জন্য তৈরি করা ব্রাঞ্চের নাম। আপনি আপনার ব্রাঞ্চের নাম আপনার ফিচার অনুসারে দিতে পারেন।

2. Switch to New Branch:
   নতুন ব্রাঞ্চে কাজ করতে যাওয়ার জন্য এই কমান্ড ব্যবহার করবে:

git checkout new-feature

এটি এনামুল কে new-feature ব্রাঞ্চে নিয়ে যাবে, এবং নতুন ফিচারটির উপর কাজ করতে পারবে।

3. Commit Your Changes:
   এখন এনামু্লের কাজ শেষ হয়ে গেলে পরিবর্তনগুলো commit করতে হবে:

git add .
git commit -m "added new feature"

4. Merge to Main Branch:
   কাজ শেষ হলে এবং ফিচারটি প্রস্তুত হলে, সেই ফিচারটি main ব্রাঞ্চে মার্জ করতে হবে। প্রথমে main ব্রাঞ্চে যাবে, এবং তারপর মার্জ করবে:

git checkout main
git merge new-feature

এখন ওদের main ব্রাঞ্চে নতুন ফিচারটি যোগ হয়ে যাবে, কোন সমস্যা ছাড়াই।

5. Delete the Feature Branch (Optional):
   যদি পরবর্তীতে মনে করেন যে ফিচারটি এখন আর প্রয়োজন নেই, তাতিত নতুন ব্রাঞ্চটি ডিলিট করতে :

git branch -d new-feature

এভাবে আপনিও গিটে branching, checkout, এবং merging এর মাধ্যমে আপনার কাজের ভার্সনগুলির মধ্যে আলাদা আলাদা ফিচার তৈরি এবং মার্জ করতে পারেন,

### Fetch, Pull & Push

আমরা এ পর্যন্ত যা যা করেছি, তা ছিলো Local Repository এর মধ্যে। আমরা যদি একটু সামারি করি, তাহলে হবে এমন:

Working Directory ==> Stage ==> Local Repository ==> Remote Repository ( যেটা এখন করবো )

এখন আমরা এই কোডটি Remote Repository-তে পাঠাবো, অর্থাৎ Cloud এ, এখানে আমরা GitHub ব্যবহার করবো।

##### Fetch

যদি রিমোট রিপোতে কোনো চেঞ্জ হয় এবং আমরা সেই পরিবর্তনগুলো আমাদের কম্পিউটারে নিয়ে আসতে চাই, তাহলে আমরা **fetch** ব্যবহার করবো, যেমন:

```bash
git fetch origin
```

##### Pull

fetch ব্যবহার করে আমরা রিমোট রিপো থেকে সর্বশেষ চেঞ্জগুলো ঠিকই আনতে পারি কিন্তু সেটা আমাদের লোকাল রিপোতে আসে না তাই সেটা করার জন্য,আমাদের **pull** ব্যবহার করবো, এতে করে সর্বশেষ চেঞ্জগুলোও আসবে একি সাথে আমার কোডও Working Directory মারজ হয়ে যাবে যেমন:

```bash
git pull origin main
```

##### Push

যদি আমরা লোকাল রিপো থেকে রিমোট রিপোজিটরিতে কোড পাঠাতে চাই, তাহলে আমরা **push** ব্যবহার করবো, যেমন:

```bash
git push origin main
```
